Overview of Fields

All electronic record systems use the concept of fields to organize information. For instance, in a typical patient database, each person has a Name, ID #, Date of Birth, etc. Most systems have a fixed set of standard fields along with a few blank spaces that can be used as custom fields.

GPD takes a flexible approach where users can define whatever fields they need for patient and treatment records just by typing information using a certain convention. For instance, to add a standard ID field to a patient's information, you just write PATIENT ID: 123456; or to add something more obscure, you could use Favorite Color: Maroon. If multiple patients (or treatments) use the same field names, then you can analyze and export data using those fields as though they were built-in.

Field Parsing

What constitutes a treatment or patient field are not predetermined, but rather collected from the textual content in the patient/treatment info text areas based on a convention: any line that starts with a letter and doesn't have two consecutive whitespace characters before the first colon, is considered a field, holding data from that line and all subsequent field-value lines.

A field-value line begins with one or more whitespace characters, and contains at least one non-whitespace character. Thus, a field's value ends when either a blank line or a new field is encountered.

An few examples of valid fields:

Age: 43

Frequency: 60 Hz
Frequency:
  60 Hz

Anesthesia: Lorazepam
            Thiopental
            Lidocaine

Reason For Study: 
  > Referral by PCP
  > Emotional Indicators

In this example, both ANESTHESIA and REASON FOR STUDY will have values composed of multiple lines.

And a few examples that are not fields:

Age of Patient - 43       (no colon)
IIE  Frequency: 67 Hz     (two spaces between IIE and Frequency)
  Patient ID:  45         (line starts with a space)
# of Visits: 4            (line does not begin with a letter)

Field names are case insensitive, so FREQUENCY and FreQueNCy are considered the same name.

In addition to all the fields derived from collected treatment data (i.e. % Energy Set, Base Heart Rate, etc.) you can define any custom fields you would like using the above convention, and these will be available for analysis in the Chart and Graph, in HL7 templates, and so forth. A common usage is to define custom fields in Patient Info – Patient ID, etc. – that required for interoperability with an installation's health records system.
